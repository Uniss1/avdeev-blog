## Контекст исследования

Продукт Tech Knowledge Base находится на стадии **пре-реализации**: существует работающий HTML-прототип (`Сайт3.html`, 473 строки) и полный набор проектных артефактов (9 документов). Реализация на Astro SSG ещё не начата. Данный отчёт оценивает:

1. UX-качество существующего прототипа
2. Соответствие плана реализации (Astro SSG) заявленным целям
3. Разрывы между прототипом и планируемой реализацией
4. Рекомендации для v1 реализации и v2 итерации

### Gap Analysis

| Разрыв | Влияние | Критичность |
|--------|---------|-------------|
| **Продукт не существует как SSG.** Весь проект — единственный HTML-файл с SPA-логикой. Ни одна из 26 запланированных задач не реализована. | Критическое — нет развёртываемого продукта. | Критическая |
| **Контент привязан к HTML.** Записи хранятся как HTML-строки в JS-объекте. Нет Markdown, нет frontmatter, нет типизации. | Высокое — невозможно масштабировать контент, нет Git-workflow для публикации. | Высокая |
| **Нет SEO-инфраструктуры.** Отсутствуют мета-теги, sitemap, robots.txt, Schema.org, canonical URL. Поисковики не могут качественно индексировать SPA. | Высокое — основной канал привлечения (поисковики) не работает. | Высокая |
| **Мобильная адаптация неполная.** В прототипе нет сворачиваемого sidebar; на мобильных sidebar занимает всю ширину экрана сверху, сжимая контент. | Среднее — мобильные пользователи получают ухудшенный опыт. | Средняя |
| **Зависимость от внешних CDN.** Google Fonts, Tailwind CSS CDN, Chart.js CDN, Lucide CDN — при недоступности сайт теряет стилизацию и функциональность. | Среднее — непредсказуемая деградация. | Средняя |

| **Пустое состояние при фильтрации по тегу не обрабатывается.** Если тег существует, но записей нет — отображается пустой список без сообщения. | Низкое — маловероятный сценарий при 3 записях. | Низкая |

## User Experience Assessment

### User Journey Walkthroughs

#### Journey 1: UC-1 — Просмотр главной страницы (Overview)

- **Проверенные шаги:** Открытие корневого URL, отображение заголовка "ОБЗОР", блок "Недавние записи", hover-эффекты (увеличение маркера, подчёркивание), клик на запись, переход на детальную страницу.
- **Результаты по прототипу (`Сайт3.html`):**
  - Заголовок "Обзор" отображается корректно (font-display, uppercase, text-5xl/7xl). Используется тег `<h2>`, что противоречит SEO-спецификации (должен быть `<h1>`).
  - Блок "Недавние записи" реализован в bujo-box. Заголовок блока **не инвертирован** (обычный bujo-header-box), хотя в UC-1 шаг 3 и в Story 3.3 указан "инвертированный заголовок" — противоречие между прототипом и спецификацией.
  - Список из 3 записей с маркерами, заголовками, датами — всё на месте.
  - Hover-эффект работает: маркер увеличивается (scale-150), заголовок подчёркивается.
  - Клик на строку вызывает `router.navigate('detail', ...)` — переход на детальную "страницу".
  - Правая колонка содержит блоки "Статистика" (Chart.js) и "Бэклог" (lined-paper) — оба исключены из scope v1.
- **Выявленные проблемы:**
  1. **Противоречие h1/h2:** В sidebar используется `<h1>` для "Индекс", а "Обзор" — `<h2>`. При реализации в Astro необходимо `<h1>` для "Обзор", а sidebar использовать `<div>`.
  2. **Альтернативный поток UC-1 4a (пустой список) не обрабатывается** — прототип не предусматривает сообщение "Записей пока нет".
  3. **Правая колонка в v1 будет пустой** — при удалении блоков "Статистика" и "Бэклог" layout потеряет визуальный баланс.
- **UX-качество:** 4/5 — Функционально полный journey. Визуальный язык последователен. Информационная архитектура интуитивна. Снижение за: противоречие с инверсией заголовка, отсутствие обработки пустого состояния, потерю визуального баланса при удалении правой колонки.

#### Journey 2: UC-2 — Просмотр списка всех записей

- **Проверенные шаги:** Переход через sidebar -> "Все записи", отображение заголовка, карточки NoteCard, сортировка по дате, hover-инверсия, клик на карточку.
- **Результаты по прототипу:**
  - Заголовок "Все записи" с border-b-4 — корректно.
  - Три карточки NoteCard: горизонтальная компоновка (дата слева, заголовок + теги справа).
  - Hover-инверсия всей карточки работает плавно (transition-colors duration-200).
  - Сортировка: записи отображаются в порядке массива `db.notes` (по id), а не по дате. Запись от 20.04 (id=3) отображается последней, хотя по дате она старше записи от 10.05 (id=2). Фактически порядок совпадает с хронологической сортировкой desc (15.05, 10.05, 20.04), но это совпадение из-за порядка элементов в массиве, а не результат явной сортировки.
  - Год "2024" захардкожен (строка 421).
  - Теги в карточках — обычный текст (`<span>`), не отдельные кликабельные ссылки. При клике на тег срабатывает переход на детальную страницу записи (потому что вся карточка — кликабельная область).
- **Выявленные проблемы:**
  1. **Теги в NoteCard не являются отдельными ссылками.** В прототипе вся карточка — один `onclick`. В Astro-реализации теги должны быть самостоятельными ссылками `<a>` на страницу фильтрации. Это требует обработки event propagation (клик по тегу не должен запускать навигацию на детальную страницу).
  2. **Год захардкожен "2024".** Уже зафиксировано в QA (MAJ-5).
  3. **Адаптивность карточек на мобильных:** В прототипе карточки переключаются в `flex-col` через md: breakpoint, но этого недостаточно — дата не получает border-bottom вместо border-right.
- **UX-качество:** 4/5 — Эффектная hover-инверсия создаёт запоминающийся визуальный паттерн. Карточки информативны. Снижение за: некликабельные теги, захардкоженный год.

#### Journey 3: UC-3 — Фильтрация записей по тегу

- **Проверенные шаги:** Клик на тег в sidebar, отображение заголовка "Тема: #<Тег>", фильтрованный список записей.
- **Результаты по прототипу:**
  - Клик на тег в sidebar вызывает `router.navigate('tag', tagName)`.
  - Заголовок формируется динамически: "Тема: #<Тег>" с border-b-4.
  - Фильтрация работает корректно: отображаются только записи, содержащие выбранный тег.
  - Те же NoteCard с hover-инверсией.
- **Выявленные проблемы:**
  1. **Пустое состояние не обрабатывается (UC-3 4a).** Если тег не совпадает ни с одной записью — пустой список без сообщения.
  2. **Теги в sidebar — `<button>`, не `<a>`.** В SPA это допустимо, но для SSG необходимы семантические ссылки `<a href="/tags/...">` для индексации поисковиками.
  3. **URL тегов не содержат slug.** В прототипе нет URL-маршрутизации для тегов — используется внутренний SPA-роутер. В SSG каждый тег станет отдельной страницей с транслитерированным slug.
  4. **Клик по тегу из карточки записи (NoteCard) невозможен.** Теги внутри карточки — не отдельные интерактивные элементы (вся карточка — один onclick).
- **UX-качество:** 3/5 — Базовая функциональность работает. Снижение за: отсутствие пустого состояния, теги в карточках не кликабельны, нет семантических ссылок.

#### Journey 4: UC-4 — Чтение записи (детальная страница)

- **Проверенные шаги:** Переход на детальную страницу, кнопка "НАЗАД", header (чёрный фон), метка даты, заголовок, теги, body с контентом.
- **Результаты по прототипу:**
  - Кнопка "НАЗАД" с иконкой arrow-left — стилистически выдержана.
  - Header: чёрный фон, белый текст, метка "Запись // DD.MM.2024", заголовок uppercase, теги с белым фоном на чёрном.
  - Body: контент отображается корректно — HTML с Tailwind-классами рендерится напрямую.
  - Запись #1 (1С Аналитика): bujo-box внутри контента, блок SQL-кода с чёрным заголовком "SQL_SNIPPET.sql".
  - Запись #2 (LLM-агенты): grid-layout 2 колонки внутри контента (стек + результат).
  - Запись #3 (Airflow): чек-лист с кастомными маркерами (X в квадрате).
- **Выявленные проблемы:**
  1. **Дата в header хардкодит год:** `${note.date}.2024` — год не извлекается из данных. В frontmatter поле `date` хранит только "DD.MM", без года.
  2. **Кнопка "НАЗАД" всегда ведёт на главную.** Пользователь, пришедший из `/tags/optimizaciya/`, ожидает возврат к фильтрованному списку, а получает главную страницу.
  3. **Нет подсветки синтаксиса.** SQL-код в `<pre>` — просто моноширинный текст. Для технического блога это значительный UX-недостаток.
  4. **Контент — сырой HTML.** При миграции в MDX потребуется либо пересоздание разметки как Markdown + MDX-компоненты, либо вставка HTML напрямую (MDX это допускает). Второй вариант проще, но создаёт XSS-риск (CRIT-3 из QA-отчёта).
  5. **Теги на детальной странице — не кликабельны.** Отображаются как `<span>`, не как `<a>`. В Astro-реализации должны стать ссылками на `/tags/[slug]/`.
  6. **Нет блока "Связанные записи"** — при чтении одной записи нет предложений для продолжения (запланировано на v2).
- **UX-качество:** 3/5 — Визуально впечатляет (инвертированный header, чёткая типографика). Контент читается хорошо. Снижение за: некликабельные теги, захардкоженный год, отсутствие подсветки кода, фиксированный "НАЗАД" на главную.

#### Journey 5: UC-5 — Навигация через sidebar

- **Проверенные шаги:** Структура sidebar, пункты навигации, облако тегов, hover-эффекты, клики по элементам.
- **Результаты по прототипу:**
  - Sidebar содержит все необходимые элементы: заголовок "ИНДЕКС", подзаголовок, блок "Навигация" (инвертированный заголовок — bg-black text-white), блок "Темы" (обычный заголовок), подвал "ПУБЛИЧНЫЙ РЕСУРС".
  - Hover на пункты навигации: `hover:font-bold` — работает.
  - Hover на теги: инверсия цветов (bg-black text-white, border-black) — работает.
  - Клик на "Обзор" — переход на home. Клик на "Все записи" — переход на all-notes. Клик на тег — переход на фильтрацию.
  - Нумерация пунктов (01, 02) — декоративная, не функциональная.
- **Выявленные проблемы:**
  1. **`<h1>` для "Индекс"** — нарушает правило "один h1 на страницу" для SEO. В Astro-реализации sidebar должен использовать `<div>` или `<p>`.
  2. **Навигация реализована через `<button onclick="...">`, не через `<a href="...">`**. В SSG необходимы семантические ссылки для индексации и доступности (клавиатурная навигация, screen readers).
  3. **Активная страница не выделена визуально.** При нахождении на странице "Все записи" пункт "Все записи" в sidebar выглядит так же, как остальные. Нет индикатора текущего раздела.
  4. **RSS-ссылка отсутствует в sidebar.** В спецификации (Story 7.1) указано: "В sidebar или footer есть иконка/ссылка на RSS-ленту".
- **UX-качество:** 4/5 — Чистая информационная архитектура. Визуальная иерархия (инвертированный заголовок "Навигация" vs обычный "Темы") интуитивна. Снижение за: отсутствие индикатора активной страницы, семантические проблемы с h1 и button/a.

#### Journey 6: UC-6 — Переход на несуществующую страницу (404)

- **Проверенные шаги:** Невозможно протестировать в прототипе.
- **Результаты:** В SPA-прототипе нет маршрута для 404. При вызове `router.navigate` с несуществующим маршрутом — ничего не произойдёт (нет обработки ошибок в роутере).
- **Оценка плана:** Спецификация (Story 7.2, UC-6) описывает полноценную страницу 404 с BaseLayout, заголовком "404", текстом, кнопкой возврата. Это адекватное решение.
- **UX-качество:** N/A (не реализовано в прототипе). По спецификации: оценка плана 4/5.

#### Journey 8: UC-8 — Индексация поисковой системой

- **Проверенные шаги:** Анализ HTML-разметки прототипа для SEO.
- **Результаты по прототипу:**
  - `<html lang="ru">` — присутствует (хорошо).
  - `<title>Knowledge Base | Tech Blog</title>` — формат не соответствует спецификации ("Обзор | Tech Knowledge Base").
  - Нет `<meta name="description">`, нет OG-тегов, нет canonical, нет sitemap, нет robots.txt.
  - SPA-архитектура не индексируется без серверного рендеринга или prerendering.
  - Контент загружается через JavaScript (`router.renderHome(container)`) — поисковые боты могут не увидеть контент.
- **Выявленные проблемы:**
  1. **SPA-прототип фактически невидим для поисковиков.** Контент генерируется через JavaScript после загрузки. Googlebot может обработать JS, но Yandexbot — нет.
  2. **Нет структурированных данных (Schema.org).** Поисковики не могут определить тип контента.
  3. **Все записи на одном URL.** Нет отдельных URL для записей — невозможна deep-link индексация.
- **UX-качество:** 1/5 — Прототип непригоден для индексации. Переход на SSG критически необходим.

#### Journey 9: UC-9 — Просмотр на мобильном устройстве

- **Проверенные шаги:** Анализ адаптивных стилей в прототипе.
- **Результаты по прототипу:**
  - Layout: `flex flex-col md:flex-row` — на мобильных sidebar отображается сверху как полноразмерная панель.
  - NoteCard: `flex flex-col md:flex-row` — на мобильных карточки перестраиваются в вертикальный layout.
  - Sidebar **не сворачивается** на мобильных — занимает значительную часть экрана, вытесняя контент.
  - Touch targets: не все элементы имеют минимальный размер 44x44px (кнопки навигации в sidebar, теги в облаке).
  - Блоки кода: `overflow-x-auto` не указан в прототипе — длинный код может выходить за границы экрана.
  - Заголовок "Обзор" на мобильных: `text-5xl` — адекватный размер.
- **Выявленные проблемы:**
  1. **Sidebar не адаптирован для мобильных.** На узких экранах он занимает всю ширину и большую высоту, заставляя пользователя скроллить вниз для доступа к контенту. Необходим сворачиваемый sidebar (details/summary или бургер-меню).
  2. **Touch targets слишком мелкие.** Теги в облаке и пункты навигации имеют padding всего 8-16px — меньше минимальных 44x44px.
  3. **Нет горизонтального скролла для кода.** `<pre>` блоки без `overflow-x-auto` выходят за пределы viewport.
- **UX-качество:** 2/5 — Базовая адаптивность через Tailwind breakpoints, но критические проблемы: несворачиваемый sidebar, мелкие touch targets, отсутствие overflow для кода.

### Accessibility Assessment

- **Клавиатурная навигация:** Частично поддерживается. В прототипе навигация реализована через `<button onclick="...">`, что делает элементы фокусируемыми через Tab. Однако теги в NoteCard — `<span>`, не фокусируемые. В Astro-реализации все интерактивные элементы должны быть `<a>` или `<button>` для полноценной клавиатурной навигации. Focus-ring (`outline: 2px solid #000; outline-offset: 2px`) не определён в прототипе — необходимо добавить в global.css.

- **Screen reader:** Слабая поддержка. `<html lang="ru">` установлен (корректная идентификация языка). Но: `<h1>` в sidebar ("Индекс") нарушает иерархию заголовков; контент генерируется через innerHTML — screen reader может не обнаружить динамический контент; нет ARIA-атрибутов. В Astro-реализации: семантическая разметка (`<nav>`, `<main>`, `<article>`, `<aside>`, `<time>`) решит большинство проблем.

- **Контрастность цветов:** Превосходная. Монохромная палитра (#000 на #FFF) обеспечивает контрастность 21:1 — существенно превышает WCAG AAA (7:1). Инвертированные элементы (#FFF на #000) — аналогично 21:1. Серые точки сетки (#d1d5db) на белом фоне — декоративные, не содержат информации. Единственный потенциальный риск: `border-white/30` (rgba(255,255,255,0.3)) в метке даты на детальной странице — контрастность 1.6:1, что ниже минимума, но это декоративный разделитель, не текст.

- **Мобильный опыт:** Неудовлетворительный. Sidebar не адаптирован (не сворачивается). Touch targets < 44px для тегов и навигации. Блоки кода без overflow-x-auto. При переходе на Astro: запланированное решение через `<details>/<summary>` и увеличение touch targets корректно адресует эти проблемы.

## Recommendations for Next Iteration

### Priority 1 (Критические для v1 реализации)

1. **Реализовать проект как Astro SSG.**
   Это фундамент всего продукта. Без SSG невозможны: индексация поисковиками, отдельные URL для записей, RSS, sitemap, zero-JS performance. План реализации (09-release-report.md) адекватен, порядок фаз оптимален.

2. **Решить проблему правой колонки на главной странице.**
   При удалении блоков "Статистика" и "Бэклог" из v1, правая колонка (col-span-1) окажется пустой. Рекомендация: перейти на layout без правой колонки (блок "Недавние записи" занимает всю ширину), либо разместить краткий блок "О блоге" с описанием и ссылками (GitHub, RSS).

3. **Реализовать сворачиваемый мобильный sidebar.**
   Текущий sidebar на мобильных занимает значительную часть экрана, ухудшая первое впечатление. Решение через `<details>/<summary>` — оптимально (zero-JS, нативная accessibility).

4. **Сделать теги кликабельными везде.**
   В NoteCard, на детальной странице записи, в sidebar — все теги должны быть ссылками `<a>` на `/tags/[slug]/`. Это критично для внутренней перелинковки и навигации.

5. **Добавить подсветку синтаксиса.**
   Для технического блога с SQL, Python, YAML — подсветка кода не "nice to have", а необходимость. Shiki (встроен в Astro) с монохромной темой — правильный выбор.

### Priority 2 (Важные для v1)

6. **Добавить визуальное выделение активной страницы в sidebar.**
   Без индикатора текущего раздела пользователь дезориентирован. Минимальное решение: `font-bold` или инверсия (bg-black text-white) для текущего пункта навигации.

7. **Обработать пустые состояния.**
   Добавить сообщения для случаев: (a) нет записей на главной, (b) нет записей с выбранным тегом. Это простое улучшение, предотвращающее пустой экран.

8. **Увеличить touch targets на мобильных.**
   Все кликабельные элементы (теги, навигация, кнопки) — минимум 44x44px. Особенно важно для облака тегов в sidebar.

9. **Добавить RSS-ссылку в UI.**
   Иконка RSS в подвале sidebar или рядом с "ПУБЛИЧНЫЙ РЕСУРС". Без видимой ссылки RSS-подписка невозможна.

10. **Определить стратегию для правой колонки на главной (v1 placeholder).**
    Если layout остаётся grid-cols-3 — правая колонка должна содержать что-то осмысленное. Варианты: краткое "Об авторе" / "О блоге", или ссылка на Concept.md, или облако тегов (дублирование sidebar, но на мобильных sidebar свёрнут).

### Priority 3 (Желательные для v1 / перспективы v2)

11. **Рассмотреть View Transitions API для v2.**
    Потеря SPA-анимаций при переходе на MPA — заметна. Astro нативно поддерживает View Transitions API, который может восстановить fade-эффект между страницами без JS-бандла.

12. **Заменить фиксированный "НАЗАД" на `history.back()` (v1.1).**
    Минимальное улучшение: inline-скрипт для `history.back()` с fallback на `/`. Восстанавливает контекст фильтрации при возврате.

13. **Добавить блок "Связанные записи" на детальной странице (v2).**
    При росте контента (10+ записей) — автоматическая подборка по общим тегам внизу детальной страницы. Улучшает engagement и время на сайте.

14. **Реализовать пагинацию для `/notes/` (v2).**
    При 30+ записях одна страница со всеми карточками станет тяжёлой. Astro поддерживает пагинацию через `paginate()`.

15. **Добавить кнопку "Копировать" для блоков кода (v2).**
    Стандартная функция для технических блогов. Требует минимального inline JS.

## Summary for Next Iteration

### Состояние продукта после итерации 1

Итерация 1 произвела **полный комплект проектной документации** (9 артефактов) и **рабочий HTML-прототип** (`Сайт3.html`). Реализация на Astro SSG **не выполнена** — продукт не развёрнут, не индексируется, не доступен публично.

Прототип демонстрирует зрелую дизайн-систему (тотальный монохром, bujo-box, инженерная геометрия) и работающие UX-паттерны (hover-инверсия, типографическая иерархия Unbounded/Geist/Geist Mono). Прототип является **эталоном визуального дизайна** для реализации.

### Ключевые выводы UX-исследования

1. **Дизайн-система — сильнейшая сторона проекта.** Уникальная визуальная идентичность, последовательная типографика, информационная плотность. Необходимо перенести 1:1 в компонентную архитектуру Astro.

2. **Прототип непригоден для публикации.** SPA-архитектура блокирует индексацию (основной канал трафика), CDN-зависимости создают хрупкость, контент в JavaScript не масштабируется.

3. **Мобильный опыт — главная UX-проблема.** Несворачиваемый sidebar, мелкие touch targets, отсутствие overflow для кода. Требует приоритетного внимания при реализации.

4. **Навигационная модель функциональна, но имеет дефекты.** Кнопка "НАЗАД" на главную (вместо history.back), некликабельные теги в карточках и на детальной странице, отсутствие индикатора активной страницы.

5. **Контент записей содержит сложную HTML-разметку.** Grid-сетки, bujo-box внутри контента, кастомные чек-листы. Миграция в чистый Markdown невозможна — необходим MDX с компонентами (`<BujoBox>`, `<CheckList>`, `<Grid>`).
