# Product Analysis: Tech Knowledge Base (Строгий Цифровой Сад)

## Executive Summary

Tech Knowledge Base — персональный технический блог в формате «Цифрового сада», предназначенный для публичной документации решений инженерных задач, архитектурных решений и результатов экспериментов. Проект отличается от типичных блогов строгой монохромной эстетикой, вдохновлённой чертежами и Bullet Journal, и ориентирован на максимальную информационную плотность без визуального шума. Уже существует рабочий HTML-прототип, демонстрирующий полную дизайн-систему и SPA-навигацию.

## Target Audience

- **Основная персона:** Автор блога — инженер данных / аналитик, работающий с 1С Аналитикой, Airflow, Python, LLM-агентами и BI-системами. Ведёт блог как публичную базу знаний для структуризации собственного опыта и формирования профессионального портфолио.
- **Вторичная персона:** Русскоязычные технические специалисты (дата-инженеры, аналитики, DevOps), которые ищут практические решения конкретных проблем через поисковые системы и наталкиваются на записи блога.
- **НЕ для:** Нетехнической аудитории. Не для массового читателя. Не для тех, кто ищет туториалы для начинающих или развлекательный контент.

## Problem Statement

Технические специалисты регулярно решают сложные задачи, но решения теряются в чатах, заметках и памяти. Существующие платформы (Medium, Habr, Dev.to) требуют соответствия формату площадки, не дают полного контроля над подачей и дизайном, перегружены рекламой и социальными механиками. Notion и Obsidian Publish решают задачу частично, но ограничивают кастомизацию и привязывают к экосистеме вендора.

Авторский статический сайт решает все эти проблемы: полный контроль над дизайном, нулевая стоимость хостинга (GitHub Pages / Netlify), контент в Markdown-файлах под Git-контролем, отсутствие зависимости от платформ.

## Core Features (v1)

### MUST (определяют MVP, без них запуск невозможен):
- **Статическая генерация сайта (SSG):** Перевод из единого HTML-файла в SSG-архитектуру (Astro, 11ty или Hugo) с контентом в Markdown/MDX-файлах. Без этого невозможно масштабировать контент.
- **Воспроизведение текущего дизайна в SSG:** Полный перенос существующей дизайн-системы (точечная сетка, bujo-box, монохромная палитра, типографика Unbounded/Geist/Geist Mono) в компонентную архитектуру выбранного генератора.
- **Четыре основных маршрута:** Главная страница (Overview/Dashboard), список всех записей, фильтрация по тегу, детальная страница записи — как в текущем прототипе.
- **Навигация:** Боковая панель с индексом и облаком тегов.
- **Адаптивная верстка:** Корректное отображение на десктопе и мобильных устройствах (текущий прототип уже содержит базовую адаптивность).
- **Деплой:** Автоматическая публикация через CI/CD (GitHub Actions) на бесплатный хостинг (GitHub Pages, Netlify или Vercel).
- **Миграция контента:** Перенос трёх существующих записей из mock-объекта `db.notes` в отдельные Markdown-файлы.

### SHOULD (важно, но не блокирует запуск):
- **Синтаксическая подсветка кода:** Интеграция Prism.js или Shiki для рендеринга блоков кода внутри записей. Критично для технического блога, но можно запуститься с базовым `<pre><code>`.
- **SEO-основы:** Мета-теги (title, description, og:image), семантическая HTML-разметка, sitemap.xml, robots.txt. Без этого блог не будет индексироваться поисковиками.
- **RSS-лента:** Для подписки через RSS-ридеры — стандартная практика для технических блогов.
- **Страница 404:** Кастомная страница ошибки в стиле дизайн-системы.

### COULD (отложить на v2):
- **Полнотекстовый поиск (Fuse.js):** Полезно при росте количества записей, при 3-10 записях — избыточно.
- **Тёмная тема (терминальный стиль):** Эффектная функция, но не влияет на ценность контента. Требует значительного объёма работ по инверсии всей дизайн-системы.
- **Дашборд со статистикой (Chart.js):** В текущем прототипе есть график роста записей. В статическом сайте данные для него нужно генерировать на этапе сборки. Откладывается как сложная и неприоритетная функция.
- **Бэклог-виджет:** Элемент «линованной бумаги» из прототипа. Интересная деталь, но не несёт ценности для читателя.
- **Визуализация метрик записей:** Поле `metrics` в данных (before/after, значения) — специализированный виджет, можно реализовать позже.
- **Анимации переходов между страницами:** В SPA-прототипе есть fade-анимация. В SSG-архитектуре это технически сложнее (View Transitions API) и не является приоритетом.

## Pre-Project Survey

### Текущее состояние кодовой базы:
- **Файловая структура:** Проект содержит 2 файла в корне: `Concept.md` (концептуальный документ, 92 строки) и `Сайт3.html` (рабочий прототип, 473 строки). Директория `docs/pipeline/` содержит `state.yaml` для управления итерациями.
- **Прототип (`Сайт3.html`):** Полностью функциональный SPA в одном HTML-файле. Включает:
  - Настроенный Tailwind CSS через CDN с кастомной конфигурацией шрифтов.
  - Полную дизайн-систему: CSS-стили для точечной сетки, bujo-box, lined-paper, кастомный скроллбар.
  - JavaScript-роутер на Vanilla JS с четырьмя маршрутами (home, all-notes, tag, detail).
  - Mock-базу данных (`db.notes`) с тремя записями, содержащими HTML-контент со встроенными Tailwind-классами.
  - Интеграцию Chart.js для графика статистики.
  - Подключение Lucide Icons.
  - Базовую адаптивность (md: breakpoints).
- **Отсутствует:** Git-репозиторий не инициализирован. Нет package.json, конфигурации сборки, CI/CD, файлов контента в Markdown, SSG-фреймворка.

### Технические ограничения:
- **Выбранные технологии:** HTML5, CSS3, Tailwind CSS, Vanilla JS. Шрифты: Unbounded, Geist, Geist Mono (Google Fonts). Иконки: Lucide. Визуализация: Chart.js.
- **Дизайн-система:** Жёстко зафиксирована — тотальный монохром (чёрный/белый), отсутствие теней и скруглений, border 2px solid, точечная сетка 24px. Любое отклонение нарушит концепцию.
- **Кириллический контент:** Весь интерфейс и записи на русском языке. SSG-фреймворк должен корректно обрабатывать кириллицу в URL, тегах и метаданных.
- **Именование файла прототипа:** Кириллическое название `Сайт3.html` — указывает на итеративный процесс прототипирования (вероятно, были версии 1 и 2).

### Внешние зависимости:
- Google Fonts CDN (Geist, Geist Mono, Unbounded).
- Tailwind CSS CDN (при переходе на SSG — npm-пакет).
- Lucide Icons CDN.
- Chart.js CDN (если будет использоваться в v1).
- Хостинг: GitHub Pages / Netlify / Vercel (бесплатный тир).
- DNS (если будет использоваться собственный домен).

### Ресурсные ограничения:
- Один разработчик с AI-ассистентом (Claude Code).
- Нет выделенного бюджета на инфраструктуру (бесплатный хостинг).
- Контент создаётся автором вручную на основе рабочего опыта.

## Assumptions

1. **Автор — единственный контрибьютор.** Нет необходимости в CMS-интерфейсе, workflow публикации или системе ролей. Контент пишется в Markdown в текстовом редакторе и коммитится в Git.
2. **Объём контента в v1 невелик (3-20 записей).** Это означает, что полнотекстовый поиск, пагинация и оптимизация времени сборки не являются приоритетами.
3. **Основной канал привлечения читателей — поисковые системы.** SEO-оптимизация важнее социальных функций (комментарии, лайки, шеры).
4. **Прототип одобрен как эталон дизайна.** Дизайн-система из `Сайт3.html` считается финальной для v1 и не требует ревизии.
5. **Tailwind CSS будет перенесён из CDN в процесс сборки** при переходе на SSG, что является стандартной практикой.
6. **HTML-контент внутри `db.notes` содержит Tailwind-классы.** При миграции в Markdown потребуется либо пересоздать разметку, либо использовать MDX для сохранения кастомных компонентов.
7. **Хостинг будет бесплатным** (GitHub Pages или аналог), поскольку статический сайт не требует серверной инфраструктуры.
8. **Кириллические URL допустимы** или будут транслитерированы — необходимо решение на этапе проектирования.

## Risks

- **Сложность миграции HTML-контента в Markdown/MDX:** Контент в прототипе содержит сложную HTML-разметку с Tailwind-классами (сетки, вложенные bujo-box, чек-листы). Простой Markdown не способен это воспроизвести. **Митигация:** Использовать MDX (Markdown + JSX-компоненты) или Shortcodes для кастомных блоков. Создать компоненты: CodeBlock, CheckList, MetricsCard, Grid.
- **Потеря SPA-поведения при переходе на SSG:** Текущий прототип — SPA с роутером и анимациями. SSG генерирует отдельные HTML-страницы. **Митигация:** Принять MPA-модель как архитектурное решение. SPA-поведение не является критичным для блога. При необходимости использовать View Transitions API (Astro поддерживает нативно).
- **Кириллица в тегах и URL:** Теги в прототипе на русском (`Оптимизация`, `Пайплайны-данных`) и смешанном (`1C-Analytics`, `AI-Агенты`). Это может вызвать проблемы с маршрутизацией и SEO. **Митигация:** Определить стратегию slug-генерации: транслитерация, латинизация или URL-кодирование. Зафиксировать в документации.
- **Зависимость от CDN (Google Fonts, Tailwind):** При недоступности CDN сайт потеряет стилизацию. **Митигация:** При переходе на SSG подключить шрифты и Tailwind через npm/локальные файлы.
- **Перфекционизм в дизайне может блокировать запуск:** Строгая дизайн-система с высокими стандартами может привести к бесконечному полированию вместо публикации контента. **Митигация:** Зафиксировать scope v1. Запуск при 3 записях. Дизайн-система уже готова в прототипе — цель v1 перенести её, а не переделывать.
- **Единственный разработчик:** Bus factor = 1. При потере мотивации проект может быть заброшен. **Митигация:** Минимальный v1 scope. Быстрый запуск для создания ощущения прогресса. Контент в Markdown + Git = данные не потеряны даже при смене SSG.

## v1 Scope Boundaries

Следующие элементы явно **ИСКЛЮЧЕНЫ** из v1:

- **Тёмная тема (терминальный стиль):** Требует создания альтернативной цветовой схемы, переключателя тем и тестирования всех компонентов в двух режимах. Откладывается до v2 после стабилизации светлой темы.
- **Полнотекстовый поиск (Fuse.js):** При 3-20 записях навигация через облако тегов и список записей покрывает потребность. Поиск станет актуален при 50+ записях.
- **Дашборд со статистикой (Chart.js):** В SSG-архитектуре данные для графика нужно генерировать на этапе сборки, что добавляет сложность. Не несёт ценности для читателя.
- **Бэклог-виджет (lined-paper):** Декоративный элемент, специфичный для автора. Не влияет на пользовательский опыт читателя.
- **Визуализация метрик записей (before/after):** Кастомные виджеты для каждой записи. Откладывается до создания системы MDX-компонентов.
- **Комментарии / обратная связь:** Не планируется. Блог — это публикация, а не платформа для дискуссий. При необходимости можно добавить ссылку на Telegram/GitHub Issues.
- **Аналитика посещений:** Можно добавить Plausible/Umami позже, но не блокирует запуск.
- **Мультиязычность (i18n):** Контент на русском языке. Перевод на английский — отдельный проект.
- **Система авторизации / черновиков:** Автор публикует через Git push. CMS-функциональность не нужна.
