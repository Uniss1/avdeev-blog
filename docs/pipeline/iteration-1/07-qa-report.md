### Attack Surface Analysis

- **GitHub репозиторий (push в main):** Высокий риск — единственная точка публикации контента. Митигация задокументирована (2FA, SSH), но не тестируется автоматически. Рекомендация: добавить Dependabot в TC или в CI чеклист.
- **npm-зависимости (supply chain):** Средний риск — `npm audit` упомянут в спецификации, но не включён в workflow CI/CD (не в deploy.yml по описанию в 05-specifications.md). Статус: не верифицировано автоматически.
- **MDX-контент (XSS):** Средний риск — MDX допускает сырой HTML, что создаёт XSS-вектор если автор случайно или намеренно вставит `<script>`. Astro санитизирует стандартный Markdown, но MDX с `set:html` или raw HTML — нет. Нет теста. Статус: неверифицированная уязвимость (CRIT-3).
- **Внешние CDN-зависимости (Яндекс Метрика, Plausible):** Средний риск — CSP задокументирован, но реализован через `<meta>` тег (менее эффективен чем HTTP-заголовок). Домены Яндекс Метрики в CSP: `mc.yandex.ru`, `mc.yandex.com`, `yastatic.net` — широкий whitelist для внешних доменов. Статус: документально покрыт, тест отсутствует.
- **GitHub Pages конфигурация (публичный репозиторий):** Низкий риск — статические файлы публичны по определению. Нет серверных секретов. Статус: приемлемо.
- **Robots.txt (доступность чувствительных URL):** Низкий риск — нет чувствительных путей. `robots.txt` разрешает всё. Статус: нормально для публичного блога.

## Existing Code Analysis (Сайт3.html)

### Качество кода

**Положительные аспекты:**
- Чистая структура SPA с явным разделением данных (`db`), маршрутизации (`router`) и рендеринга.
- Дизайн-система полностью реализована через CSS-классы и последовательно применена.
- Tailwind-конфигурация корректно расширяет шрифты и `borderWidth`.

**Потенциальные проблемы:**

1. **XSS через innerHTML (строки 284, 287, 314, 415, 456):** Рендерер использует `container.innerHTML = \`...\`` и вставляет данные из `db.notes` напрямую: `${note.title}`, `${note.content}`, `${note.tags.map(t => ...)}`. Поле `note.content` содержит HTML с Tailwind-классами — это намеренно, но в контексте Astro-миграции это означает что MDX-компоненты должны заменить прямой HTML. При миграции, если MDX разрешает сырой HTML, риск сохраняется.

2. **Захардкоженный год "2024" (строка 422):** `<span>2024</span>` — не динамически из данных. Уже указано в MAJ-5. В `db.notes` у записей поле `date` хранит только "DD.MM" без года, что означает что год нигде не хранится в данных.

3. **Нет обработки ошибок в роутере:** `router.navigate('detail', data)` не проверяет что `data !== null`. Если передать `null` — `renderDetail` получит `null.title` и бросит TypeError. В SSG-версии это не проблема, но свидетельствует о хрупкости прототипа.

4. **Chart.js инициализируется через `setTimeout` (строка 361):** `setTimeout(() => { ... }, 50)` — хак для ожидания рендеринга DOM. В SSG (Astro) Chart.js исключён из v1, но этот паттерн показывает техдолг при возможном возвращении.

5. **`lucide.createIcons()` вызывается в `navigate()` после каждого рендера (строка 275):** Это CDN-зависимость. В Astro-версии используется npm-пакет `lucide-astro` с инлайнингом SVG, что архитектурно лучше.

6. **Отсутствие `<meta name="description">` и SEO-тегов:** Прототип имеет только `<title>Knowledge Base | Tech Blog</title>` — без description, OG-тегов, canonical. Это нормально для прототипа, но должно быть устранено при миграции.

7. **Несоответствие title в прототипе:** `<title>Knowledge Base | Tech Blog</title>` vs. ожидаемый формат в спецификации `"Обзор | Tech Knowledge Base"`. Форматы разные: порядок слов, разный разделитель (|), разное название.

8. **Теги в sidebar не имеют `href`-ссылок (строка 285):** Используются `<button onclick="router.navigate('tag', '${tag}')">` вместо `<a href="/tags/...">`. При SPA это нормально, но в SSG нужны настоящие ссылки для SEO (теги должны быть краулируемы поисковиками). Архитектура корректно это решает через `TagBadge`.

9. **Нет обработки пустого тега в `renderList` (строка 405):** Если `filterTag` существует, но ни одна запись не содержит этого тега, отображается пустой список без сообщения. UC-3 альтернативный поток 4a требует: "Нет записей по теме #<Тег>". Этот случай не обрабатывается в прототипе. Необходимо реализовать в Astro-версии.

## Recommendations

1. **[CRIT] Добавить TC для проверки содержимого `deploy.yml` на соответствие security requirements.** Создать тест-кейс TC-41: grep/parse `deploy.yml` и проверить: наличие `permissions:` с минимальными правами, использование `npm ci`, отсутствие `@latest` в actions. Тип: unit (проверка файла).

2. **[CRIT] Добавить TC для XSS-тестирования MDX-контента.** Создать TC-42: поместить `<script>alert('xss')</script>` в MDX-файл и убедиться что Astro/remark его эскейпит. Добавить в CI как обязательный build-тест.

3. **[CRITICAL] Устранить противоречие с количеством `<h1>` на странице.** Провести аудит: в прототипе `<h1>` для "Индекс" в sidebar означает что sidebar `<h1>` существует на каждой странице. Заменить на `<p>` или `<div>` с явной стилизацией, зарезервировав `<h1>` для основного контента страницы.

4. **[MAJOR] Добавить тест-кейс TC-43 для нефункциональных требований NFR-5 и NFR-14.** TC-43: в CI после сборки выполнять `time npm run build` и проверять что время < 60 секунд. TTFB добавить в TC-36 через инструмент curl или WebPageTest.

5. **[MAJOR] Исправить захардкоженный год в NoteCard.** Заменить строковый "2024" на динамическое `note.data.date.getFullYear()` как в Astro-компоненте, так и в обновлённом TC-19.

6. **[MAJOR] Добавить обработку пустого состояния для страниц тегов.** В `TagPage` (`/tags/[tag]/`) при нулевом количестве записей отображать сообщение "Нет записей по теме #<Тег>". Добавить TC-44 для этого граничного случая.

7. **[MAJOR] Включить `npm audit` в CI pipeline.** В `deploy.yml` добавить шаг `npm audit --audit-level=high` перед `npm run build`. Это устранит неверифицированный A06 (Vulnerable Components) из OWASP.

8. **[MINOR] Уточнить в спецификации обработку часового пояса для дат.** В схеме Content Collection использовать `z.string().transform(s => new Date(s + 'T00:00:00'))` для предотвращения UTC off-by-one ошибки при отображении дат в российском часовом поясе. Обновить TC-3.

9. **[MINOR] Добавить проверку OG-тегов и Schema.org в существующие тест-кейсы.** Расширить TC-28 и TC-29: добавить проверку `og:locale`, `og:site_name`, Schema.org WebSite на главной, Schema.org BreadcrumbList на детальной странице.

10. **[MINOR] Документировать ограничение "НАЗАД" ведёт только на главную.** В UX-прототипе и tech-specs явно зафиксировать: кнопка "НАЗАД" всегда ведёт на `/`, а не на предыдущую страницу. Добавить это в Known Limitations для v1, запланировать `history.back()` на v2.

11. **[MINOR] Добавить тест на отображение нулевых записей на главной.** TC-45: запустить сайт с пустой директорией `src/content/notes/` и убедиться что главная страница корректно отображает "Записей пока нет" вместо пустого блока или ошибки.

12. **[INFO] Рассмотреть Dependabot для автоматического обнаружения уязвимостей.** 05-specifications.md упоминает Dependabot, но нет acceptance criteria и TC. Добавить в acceptance criteria Story 1.3: файл `.github/dependabot.yml` создан.
