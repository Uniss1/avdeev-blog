# Architecture: Tech Knowledge Base (Строгий Цифровой Сад)

## Application Architecture

### System Overview

Tech Knowledge Base — статический сайт (SSG), генерируемый фреймворком **Astro** из Markdown/MDX-контента. Система не имеет серверного бэкенда, базы данных или API в традиционном смысле. Весь контент хранится в Git-репозитории в виде Markdown-файлов с frontmatter-метаданными. На этапе сборки Astro читает контент, применяет компоненты и шаблоны, генерирует набор статических HTML/CSS/JS-файлов, которые деплоятся на GitHub Pages.

**Границы системы:**
- **Вход:** Markdown-файлы в репозитории (контент), Astro-компоненты (представление), `tailwind.config.mjs` (дизайн-токены)
- **Процесс:** Сборка Astro (`astro build`) — парсинг Markdown, валидация frontmatter (Zod), рендеринг компонентов, генерация HTML, обработка CSS (Tailwind), генерация sitemap/RSS
- **Выход:** Директория `dist/` со статическими файлами → GitHub Pages
- **Клиент:** Браузер пользователя, получающий статические файлы через CDN GitHub Pages

Система намеренно не содержит клиентского JavaScript (zero-JS по умолчанию в Astro). Все интерактивные эффекты (hover, инверсия) реализованы на чистом CSS. Единственное исключение — опциональный inline-скрипт для мобильного бургер-меню.

### Components

#### Уровень 1: Инфраструктура сборки

1. **Astro Build Pipeline**
   - Ответственность: Оркестрация всего процесса сборки — от чтения Markdown-файлов до генерации статического сайта
   - Технология: Astro 5.x, Node.js 20+
   - Интерфейсы:
     - Вход: `src/` директория (компоненты, контент, стили, конфигурация)
     - Выход: `dist/` директория (статические файлы для деплоя)
     - Конфигурация: `astro.config.mjs`

2. **Content Collections (Astro)**
   - Ответственность: Типизированный доступ к Markdown-контенту с валидацией frontmatter через Zod-схемы. Обеспечивает type-safety на этапе сборки.
   - Технология: Astro Content Collections API, Zod
   - Интерфейсы:
     - Вход: `src/content/notes/*.md` (или `.mdx`)
     - Конфигурация: `src/content/config.ts` (Zod-схема)
     - Выход: Типизированные объекты записей, доступные через `getCollection('notes')`

3. **Tailwind CSS Engine**
   - Ответственность: Генерация оптимизированного CSS из utility-классов, используемых в компонентах. Включает кастомные дизайн-токены (шрифты, border-width).
   - Технология: Tailwind CSS 3.x (через `@astrojs/tailwind`), `@tailwindcss/typography` (prose plugin)
   - Интерфейсы:
     - Конфигурация: `tailwind.config.mjs`
     - Вход: CSS-классы в `.astro`-компонентах
     - Выход: Оптимизированный CSS-файл в `dist/`

4. **CI/CD Pipeline**
   - Ответственность: Автоматическая сборка и деплой сайта при push в ветку `main`
   - Технология: GitHub Actions
   - Интерфейсы:
     - Триггер: push в `main`
     - Процесс: checkout → install → build → deploy
     - Выход: обновлённые файлы на GitHub Pages

#### Уровень 2: Layout-компоненты

5. **BaseLayout (`src/layouts/BaseLayout.astro`)**
   - Ответственность: Корневой layout всех страниц. Определяет HTML-скелет (`<html>`, `<head>`, `<body>`), подключает глобальные стили, шрифты, задаёт двухколоночную структуру (sidebar + main).
   - Технология: Astro-компонент
   - Интерфейсы:
     - Props: `title` (string), `description` (string, опционально), `ogImage` (string, опционально)
     - Slot: основное содержимое страницы (вставляется в `<main>`)
   - Содержит: компоненты Sidebar, SEOHead

6. **Sidebar (`src/components/Sidebar.astro`)**
   - Ответственность: Боковая панель навигации. Отображает заголовок сайта, навигационные ссылки, облако тегов, подвал. На десктопе — фиксированная колонка w-72, на мобильных — адаптивная (горизонтальная панель с бургером или сворачиваемая).
   - Технология: Astro-компонент
   - Интерфейсы:
     - Данные: получает список всех тегов из Content Collections на этапе сборки
     - Навигация: ссылки `<a>` на `/`, `/notes/`, `/tags/[slug]/`

7. **SEOHead (`src/components/SEOHead.astro`)**
   - Ответственность: Генерация мета-тегов в `<head>`: `<title>`, `description`, Open Graph, `lang`, каноническая ссылка.
   - Технология: Astro-компонент
   - Интерфейсы:
     - Props: `title`, `description`, `ogTitle`, `ogDescription`, `ogImage`, `canonicalUrl`

#### Уровень 3: Страничные компоненты

8. **HomePage (`src/pages/index.astro`)**
   - Ответственность: Главная страница "Обзор". Отображает заголовок и список недавних записей (до 10, сортировка по дате).
   - Технология: Astro-страница
   - Интерфейсы:
     - Данные: `getCollection('notes')`, сортировка по `date` desc
     - Рендер: Grid layout (2/3 + 1/3 на десктопе), список записей с маркерами

9. **AllNotesPage (`src/pages/notes/index.astro`)**
   - Ответственность: Страница "Все записи". Отображает полный каталог записей через компонент NoteCard.
   - Технология: Astro-страница
   - Интерфейсы:
     - Данные: `getCollection('notes')`, сортировка по `date` desc
     - Рендер: вертикальный список NoteCard

10. **TagPage (`src/pages/tags/[tag].astro`)**
    - Ответственность: Страница фильтрации по тегу. Генерируется статически для каждого уникального тега.
    - Технология: Astro-страница с динамическим маршрутом
    - Интерфейсы:
      - `getStaticPaths()`: генерирует пути для всех уникальных тегов
      - Данные: отфильтрованный `getCollection('notes')` по тегу
      - Рендер: заголовок "Тема: #Тег" + список NoteCard

11. **NoteDetailPage (`src/pages/notes/[...slug].astro`)**
    - Ответственность: Детальная страница записи. Отображает полный контент одной Markdown/MDX-записи.
    - Технология: Astro-страница с динамическим маршрутом
    - Интерфейсы:
      - `getStaticPaths()`: генерирует пути для всех записей
      - Данные: одна запись из Content Collection + рендеринг Markdown
      - Рендер: BackButton, NoteHeader (чёрный фон), prose-контент

12. **NotFoundPage (`src/pages/404.astro`)**
    - Ответственность: Кастомная страница 404 в стиле дизайн-системы.
    - Технология: Astro-страница
    - Интерфейсы:
      - Рендер: заголовок "404", текст, кнопка возврата

#### Уровень 4: UI-компоненты (переиспользуемые)

13. **BujoBox (`src/components/BujoBox.astro`)**
    - Ответственность: Универсальный контейнер с рамкой 2px solid black и белым фоном. Базовый строительный блок дизайн-системы.
    - Технология: Astro-компонент
    - Интерфейсы:
      - Props: `title` (string, опционально), `headerVariant` ('default' | 'inverted'), `class` (string, опционально)
      - Slot: содержимое контейнера

14. **NoteCard (`src/components/NoteCard.astro`)**
    - Ответственность: Карточка записи для списков. Горизонтальная компоновка: блок даты слева + заголовок и теги справа. Вся карточка — ссылка.
    - Технология: Astro-компонент
    - Интерфейсы:
      - Props: `title` (string), `date` (Date), `tags` (string[]), `slug` (string)
      - Рендер: `<a>` элемент на `/notes/[slug]/`

15. **TagBadge (`src/components/TagBadge.astro`)**
    - Ответственность: Отображение одного тега как кликабельного бейджа. Ссылка на страницу фильтрации по тегу.
    - Технология: Astro-компонент
    - Интерфейсы:
      - Props: `tag` (string), `tagSlug` (string), `variant` ('default' | 'inverted')
      - Рендер: `<a>` на `/tags/[tagSlug]/`

16. **BackButton (`src/components/BackButton.astro`)**
    - Ответственность: Кнопка "НАЗАД" для детальных страниц. Стилизована в bujo-стиле с иконкой arrow-left.
    - Технология: Astro-компонент
    - Интерфейсы:
      - Props: `href` (string, по умолчанию `/`)
      - Рендер: `<a>` со стилями border-2, hover:bg-black

#### Уровень 5: Утилиты

17. **Slug-утилита (`src/utils/transliterate.ts`)**
    - Ответственность: Транслитерация кириллических строк в латиницу для URL-safe slug'ов. Используется для генерации slug'ов тегов.
    - Технология: TypeScript-функция (чистая, без зависимостей)
    - Интерфейсы:
      - Вход: `string` (кириллическая строка, напр. `"Оптимизация"`)
      - Выход: `string` (латинский slug, напр. `"optimizaciya"`)
    - Примечание: slug записей определяется именем файла (Astro Content Collections default), а slug тегов генерируется этой утилитой.

18. **Date-утилита (`src/utils/date.ts`)**
    - Ответственность: Форматирование дат для отображения в UI (DD.MM, DD.MM.YYYY, и т.д.).
    - Технология: TypeScript-функция
    - Интерфейсы:
      - Вход: `Date` объект
      - Выход: форматированная строка

### Data Model

```
Note (Content Collection: "notes")
  - title: string (обязательный) — заголовок записи, кириллица
  - date: Date (обязательный) — дата публикации
  - category: string (обязательный) — категория: "BI", "AI", "Инженерия" и т.д.
  - tags: string[] (обязательный) — массив тегов, кириллица/латиница
  - type: string (обязательный) — тип записи: "Техническое", "Эксперимент", "Лучшая практика"
  - description: string (опциональный) — краткое описание для SEO meta description
  - metrics: MetricsData | null (опциональный) — данные для виджета метрик (v2)
  → slug: генерируется из имени файла (Astro convention)
  → content: тело Markdown/MDX файла

MetricsData (встроенный тип, не отдельная сущность)
  - label: string — название метрики
  - before: number | null — значение "до"
  - after: number | null — значение "после"
  - value: number | null — абсолютное значение (если нет before/after)

Tag (виртуальная сущность — не хранится, вычисляется на этапе сборки)
  - name: string — оригинальное написание тега (из frontmatter)
  - slug: string — транслитерированный slug для URL
  - count: number — количество записей с этим тегом
  → has_many: Note (через фильтрацию)

SiteConfig (статическая конфигурация в astro.config.mjs и/или src/consts.ts)
  - siteName: string — "Tech Knowledge Base"
  - siteDescription: string — описание для SEO
  - siteUrl: string — базовый URL сайта
  - author: string — имя автора
```

### API Design (маршруты / страницы)

Поскольку это статический сайт, "API" в данном контексте — это маршруты (URL), генерируемые на этапе сборки.

| Маршрут | Файл-источник | Назначение | Входные данные | Генерируемый контент |
|---------|--------------|-----------|----------------|---------------------|
| `/` | `src/pages/index.astro` | Главная страница "Обзор" | `getCollection('notes')`, сортировка по дате | HTML: заголовок + список недавних записей (до 10) |
| `/notes/` | `src/pages/notes/index.astro` | Список всех записей | `getCollection('notes')`, сортировка по дате | HTML: заголовок + карточки NoteCard всех записей |
| `/notes/[slug]/` | `src/pages/notes/[...slug].astro` | Детальная страница записи | Одна запись из Content Collection | HTML: header (чёрный) + отрендеренный Markdown-контент |
| `/tags/[tag]/` | `src/pages/tags/[tag].astro` | Записи, отфильтрованные по тегу | Отфильтрованная коллекция по tag slug | HTML: заголовок "Тема: #Тег" + карточки NoteCard |
| `/404` | `src/pages/404.astro` | Страница ошибки | — | HTML: "404" + ссылка на главную |
| `/rss.xml` | `src/pages/rss.xml.ts` | RSS-лента | `getCollection('notes')` | XML: RSS 2.0 feed со всеми записями |
| `/sitemap-index.xml` | (генерируется `@astrojs/sitemap`) | Карта сайта для поисковиков | Все маршруты | XML: sitemap |
| `/robots.txt` | `public/robots.txt` | Инструкции для поисковых ботов | — | Текст: Allow all + ссылка на sitemap |

## Server Architecture

### Deployment Topology

```
[Разработчик (локальная машина)]
        |
        | git push main
        ↓
[GitHub Repository] → [GitHub Actions CI/CD]
                              |
                              | npm ci → npm run build → deploy
                              ↓
                      [GitHub Pages CDN]
                              |
                              | HTTPS
                              ↓
                      [Браузер пользователя]
```

- **Среда разработки:** Локальная машина разработчика (WSL2/Linux). Node.js 20+, npm.
- **Среда CI/CD:** GitHub Actions runner (ubuntu-latest). Запускается при push в `main`.
- **Среда продакшена:** GitHub Pages — бесплатный статический хостинг с CDN. Автоматический HTTPS.
- **DNS (опционально, v2):** Если потребуется собственный домен — настройка CNAME через GitHub Pages settings.

### Конфигурация окружений

| Параметр | Development | Production |
|----------|-------------|------------|
| URL | `http://localhost:4321` | `https://<username>.github.io/<repo>/` |
| Сборка | `astro dev` (HMR, без оптимизации) | `astro build` (оптимизация, tree-shaking) |
| CSS | Полный Tailwind (все классы) | Purged Tailwind (только используемые) |
| Source maps | Включены | Отключены |
| Base path | `/` | `/<repo>/` (настраивается в `astro.config.mjs`) |

### Infrastructure Requirements

#### Compute
- **Локальная разработка:** Любая машина с Node.js 20+. Минимальные требования: 2 GB RAM, 1 CPU. Astro dev server потребляет ~100-200 MB RAM.
- **CI/CD:** GitHub Actions runner (ubuntu-latest). Бесплатные 2000 минут/месяц для публичных репозиториев, 500 минут для приватных. Одна сборка Astro-сайта с 3-20 записями занимает ~30-60 секунд.
- **Продакшен:** Нет серверных вычислений. Только статические файлы на CDN.

#### Storage
- **Репозиторий:** GitHub — бесплатно до 5 GB. При 3-20 Markdown-записях и Astro-проекте — ~50-100 MB (включая node_modules в .gitignore, шрифты в public/).
- **GitHub Pages:** Лимит 1 GB на сайт. Статический Astro-сайт с 20 записями — ~5-15 MB. Запас значительный.
- **Шрифты (self-hosted):** ~500 KB - 1 MB (woff2-файлы Geist, Geist Mono, Unbounded).

#### Network
- **GitHub Pages CDN:** Автоматический HTTPS, HTTP/2, глобальный CDN от Fastly. Пропускная способность: 100 GB/месяц (soft limit). Для персонального технического блога — с запасом на порядки.
- **Отсутствие серверных запросов:** Нет API-вызовов, нет баз данных, нет WebSocket. Браузер получает статические файлы и работает полностью автономно.

## Technical Risks

| Риск | Влияние | Вероятность | Митигация |
|------|---------|-------------|-----------|
| **Сложность миграции HTML-контента в Markdown.** Контент в `db.notes` содержит сложную HTML-разметку с Tailwind-классами (grid, bujo-box, чек-листы с кастомными маркерами). Чистый Markdown не может это воспроизвести. | Высокое — невозможность передать визуальную структуру записей. Потеря внешнего вида, одобренного в прототипе. | Высокая — все три записи используют кастомную HTML-разметку. | 1) Использовать MDX вместо чистого Markdown для записей с кастомной вёрсткой. MDX позволяет импортировать Astro/React-компоненты прямо в контент. 2) Создать набор MDX-компонентов: `<BujoBox>`, `<CheckList>`, `<CodeBlock>`, `<Grid>`. 3) Для простых записей (будущих) — чистый Markdown достаточен. MDX нужен только для записей с кастомной визуальной структурой. 4) Fallback: оставить HTML-разметку в `.mdx`-файлах напрямую — MDX это допускает. |
| **Транслитерация тегов — неоднозначность и коллизии.** Разные кириллические строки могут дать одинаковый slug. Нет общепринятого стандарта транслитерации (ГОСТ, BGN/PCGN, произвольная). | Среднее — некорректные URL, битые ссылки, коллизии тегов. | Низкая — при 6-8 уникальных тегах коллизия маловероятна. | 1) Выбрать и зафиксировать таблицу транслитерации (ГОСТ 7.79-2000 система Б — наиболее привычна русскоязычным пользователям). 2) Написать unit-тесты для всех существующих тегов. 3) В будущем: хранить slug в frontmatter явно, а не генерировать автоматически. |
| **GitHub Pages base path.** При деплое на `https://<user>.github.io/<repo>/` все ассеты и ссылки должны учитывать base path (`/<repo>/`). Забытый base path приводит к 404 на все ресурсы. | Высокое — сайт визуально сломан или ссылки не работают. | Средняя — типичная ошибка при первом деплое на GitHub Pages. | 1) Настроить `base` и `site` в `astro.config.mjs` с самого начала (Story 1.1). 2) Использовать хелпер Astro `import.meta.env.BASE_URL` для всех ссылок. 3) Тестировать сборку через `astro preview` перед деплоем. 4) Альтернатива: деплоить как `<user>.github.io` (без подпути) если репозиторий назван `<user>.github.io`. |
| **Нулевой JS vs интерактивность sidebar на мобильных.** Astro генерирует страницы без JS. Бургер-меню или сворачиваемый sidebar требует либо JS, либо CSS-хака (checkbox/details). CSS-решения имеют ограничения UX (accessibility, анимации). | Низкое — desktop-версия работает идеально. Мобильная навигация — ухудшенный, но рабочий опыт. | Средняя — мобильная аудитория для технического блога на русском невелика, но адаптивность заявлена как MUST. | 1) Первичная реализация: `<details>/<summary>` на чистом CSS — простейший вариант, accessibility из коробки. 2) Если UX недостаточен — добавить inline `<script>` (10-20 строк) для toggle sidebar. `is:inline` директива Astro. 3) Не использовать React/Vue только ради бургер-меню. |
| **Размер self-hosted шрифтов.** Три семейства шрифтов в нескольких начертаниях могут весить 500 KB - 1.5 MB. Для статического сайта это значительная часть payload. | Низкое — увеличение First Contentful Paint на медленных соединениях. | Средняя — зависит от количества подключаемых начертаний. | 1) Подключать только реально используемые начертания: Geist (400, 500, 700), Geist Mono (400, 700), Unbounded (700, 900). Это ~6 файлов × ~50-80 KB = ~300-500 KB. 2) Использовать только woff2 (максимальное сжатие, поддержка 97%+ браузеров). 3) `font-display: swap` для предотвращения FOIT. 4) Preload критичных шрифтов через `<link rel="preload">`. |
| **Astro major version breaking changes.** Astro активно развивается. При обновлении с 4.x на 5.x возможны breaking changes в Content Collections API, конфигурации, интеграциях. | Среднее — необходимость рефакторинга при обновлении. | Низкая — для v1 достаточно зафиксировать версию. | 1) Зафиксировать точную версию Astro в `package.json` (не `^`). 2) Использовать `package-lock.json` для reproducible builds. 3) Обновлять Astro только когда есть необходимость, не автоматически. |

## Architecture Decisions

| Решение | Рассмотренные варианты | Выбор | Обоснование |
|---------|----------------------|-------|-------------|
| **SSG-фреймворк** | 1) Astro 2) 11ty (Eleventy) 3) Hugo 4) Next.js (static export) | **Astro 5.x** | Astro оптимален по совокупности: (1) нативные Content Collections с Zod-валидацией, (2) zero-JS по умолчанию — идеально для статического блога, (3) встроенная поддержка MDX для кастомных компонентов в контенте, (4) Tailwind CSS интеграция из коробки, (5) встроенный Shiki для подсветки кода, (6) View Transitions API для будущих анимаций (v2), (7) активная экосистема и документация. 11ty проще, но не имеет типизации и MDX. Hugo быстрее, но Go-шаблоны ограничены для кастомных компонентов. Next.js избыточен для статического блога. |
| **Формат контента** | 1) Чистый Markdown (.md) 2) MDX (.mdx) 3) HTML в Astro-страницах | **MDX (.mdx) для записей с кастомной вёрсткой, Markdown (.md) для простых записей** | Контент в прототипе содержит сложную HTML-разметку (grid, bujo-box, чек-листы). Чистый Markdown не может это воспроизвести. MDX позволяет использовать Astro/JSX-компоненты прямо в Markdown-файлах: `<BujoBox>`, `<CheckList>`, `<Grid>`. Для будущих простых записей (текст + код) достаточно чистого Markdown. Content Collection поддерживает оба формата одновременно. |
| **Стратегия slug-генерации** | 1) Транслитерация (ГОСТ) 2) Ручные slug в frontmatter 3) URL-кодирование кириллицы 4) Английские slug (перевод) | **Транслитерация + возможность переопределения через frontmatter** | Транслитерация даёт читаемые и SEO-friendly URL. URL-кодирование (`%D0%9E%D0%BF%D1%82...`) нечитаемо и длинно. Ручные slug создают бремя для автора. Английские slug требуют перевода — дополнительная работа. Решение: slug записей генерируется из имени файла (Astro default — файлы именуются в транслите), slug тегов генерируется утилитой `transliterate()`. |
| **Подключение шрифтов** | 1) Google Fonts CDN 2) @fontsource npm-пакеты 3) Self-hosted файлы в public/fonts/ | **Self-hosted файлы в `public/fonts/`** | Устраняет внешнюю зависимость (Google Fonts CDN). Улучшает privacy (нет запросов к Google). Даёт полный контроль над subset и формат файлов. Проще отладка (файлы в репозитории). @fontsource — допустимая альтернатива, но добавляет npm-зависимости для шрифтов, что менее прозрачно. |
| **Хостинг** | 1) GitHub Pages 2) Netlify 3) Vercel 4) Cloudflare Pages | **GitHub Pages** | (1) Бесплатен и не требует дополнительных аккаунтов — код уже на GitHub. (2) Минимальная конфигурация через GitHub Actions. (3) Автоматический HTTPS. (4) Достаточен для статического блога с малым трафиком. Netlify/Vercel дают больше функций (серверные функции, формы, аналитика), но для статического блога это избыточно. Cloudflare Pages — хорошая альтернатива, но добавляет ещё один сервис. |
| **CSS-фреймворк** | 1) Tailwind CSS 3.x 2) Чистый CSS 3) Tailwind CSS 4.x | **Tailwind CSS 3.x** | Прототип уже использует Tailwind-классы — миграция 1:1. Tailwind 3.x стабилен и хорошо поддерживается Astro-интеграцией. Tailwind 4.x (вышел в 2025) имеет breaking changes в конфигурации и ещё не все интеграции обновлены. Чистый CSS потребует переписывания всех классов из прототипа. Решение: начать с Tailwind 3.x, мигрировать на 4.x в будущем при необходимости. |
| **Подсветка синтаксиса** | 1) Shiki (встроен в Astro) 2) Prism.js 3) Highlight.js | **Shiki** | Встроен в Astro — нулевая конфигурация для базовой работы. Более точная подсветка (основан на TextMate грамматиках, как в VS Code). Рендерится на этапе сборки — нулевой JS на клиенте. Поддерживает кастомные темы. Prism.js требует подключения через плагин и добавляет JS на клиент. |
| **Иконки** | 1) lucide-astro (npm) 2) Lucide CDN (как в прототипе) 3) SVG-файлы вручную 4) astro-icon | **lucide-astro npm-пакет** | Интеграция через Astro-компоненты (`<ArrowLeft />`). Нулевой JS — SVG инлайнится на этапе сборки. Type-safety (TypeScript автокомплит по именам иконок). CDN требует клиентского JS для `lucide.createIcons()`. Ручные SVG — бремя поддержки. |
| **Мобильная навигация** | 1) CSS `<details>/<summary>` 2) Inline JS toggle 3) CSS checkbox hack 4) Astro island (React/Preact) | **CSS `<details>/<summary>` с fallback на inline JS** | Zero-JS решение. Нативная accessibility (keyboard, screen readers). Простая реализация. Ограничение: анимация раскрытия требует CSS-трюков или JS. Если UX недостаточен — добавить 15-20 строк inline JS (`is:inline`). Astro island (React) — чрезмерная зависимость ради одной кнопки. |
| **TypeScript** | 1) Strict TypeScript 2) JavaScript без типов 3) JSDoc-типы | **TypeScript (strict)** | Astro нативно поддерживает TypeScript. Strict mode ловит ошибки на этапе сборки (невалидные props, отсутствующие поля frontmatter). Content Collections используют Zod, который тесно интегрирован с TypeScript. Для проекта одного разработчика строгая типизация — заменитель code review. |

## Допущения (зафиксированные в архитектуре)

1. **Astro 5.x** — текущая стабильная версия на момент проектирования (февраль 2026). Content Collections API стабилен начиная с Astro 2.0.
2. **Tailwind CSS 3.x** — используется через `@astrojs/tailwind` интеграцию. Миграция на Tailwind 4.x не планируется в v1.
3. **Node.js 20 LTS** — минимальная версия для Astro 5.x и GitHub Actions runners.
4. **Публичный репозиторий на GitHub** — для бесплатного GitHub Pages и GitHub Actions.
5. **Одна языковая версия (русский)** — i18n не требуется. `<html lang="ru">` зафиксирован.
6. **Масштаб контента v1: 3 записи, 6-8 уникальных тегов.** Архитектура рассчитана на рост до 100+ записей без изменений.
7. **Единственный разработчик + AI-ассистент.** Нет необходимости в branch protection, PR reviews, staging environments.
